### Key Architecture Decisions

- `SessionStore` is the unified persistence interface — in-memory default (`createSessionStore`), file-backed (`createFileStore`), pluggable backend (SQLite, Redis) goes here when needed
- `PauseManager` uses `EventEmitter` internally — compatible with Node 22
- Crash detector uses heartbeat keys `hb:<sessionId>` with `Date.now()` timestamps; threshold configurable per `detectStaleSessions()` call
- Validator produces `warnings[]` not `errors[]` for role alternation violations — non-fatal heuristics
- Reducer is pure function with no side effects

### Known Gaps / Risks

1. No E2E tests for CLI session commands — these use `@microsoft/tui-test` and need test infrastructure
2. File store is synchronous JSON read/write — fine for CLI, swap for SQLite if performance matters
3. No runtime `SessionManager` integration yet — `@agentsy/runtime` still uses raw `SessionStore` primitives
4. `@agentsy/types` `RuntimeSnapshot` has slightly different shape than `CheckpointSchema` — minor alignment needed
