# @agentsy Standalone Packages ŌĆö Renderers & UI Extension Path

![Status: Planned](https://img.shields.io/badge/status-Planned-blue)

This plan ensures that all stream-parsing and rendering components of the current `@selfagency/llm-stream-parser` package remain available as independently installable, framework-agnostic packages that third parties can use without pulling in any of the `@agentsy` orchestration layer (`@agentsy/agent`, `@agentsy/memory`, `@agentsy/mcp`, etc.).

It also extends the `@agentsy` architecture to support graphical UI targets ŌĆö not just terminal UIs ŌĆö and defines the package foundation for future product-layer applications: a VS Code Copilot chat participant extension and a Claude Desktop-style desktop application.

This plan is additive to `agentsy-platform-v2.md`. New identifiers start at REQ-110, SEC-028, CON-024, GUD-021, PAT-018, ADR-066.

## 1. Requirements & Constraints

### Functional Requirements

- **REQ-110**: `@agentsy/renderers` MUST be a first-class package in the monorepo, independently installable with no dependencies on `@agentsy/agent`, `@agentsy/memory`, `@agentsy/mcp`, `@agentsy/runtime`, or any orchestration package. Its only `@agentsy/*` dependency is `@agentsy/processor` (transitively: `@agentsy/core` + `@agentsy/normalizers`).
- **REQ-111**: `@agentsy/renderers` MUST export each renderer on a dedicated subpath to enable per-renderer tree-shaking: `@agentsy/renderers` (types only), `@agentsy/renderers/plain`, `@agentsy/renderers/cli`, `@agentsy/renderers/vscode`, `@agentsy/renderers/browser`, `@agentsy/renderers/ink`. All peer dependencies remain opt-in.
- **REQ-112**: `@agentsy/renderers/vscode` MUST remain duck-typed ŌĆö no hard dependency on the `vscode` module at runtime. The `ChatResponseStream` and `CancellationToken` interfaces are structurally typed. VS Code extension authors pass their own `vscode.ChatResponseStream` instance; the renderer accepts it without importing the SDK.
- **REQ-113**: `@agentsy/renderers` MUST expose a `DisplayPort` interface: a generic output abstraction that renderer implementations can target as an alternative to direct platform output. `DisplayPort` decouples renderer logic from its delivery mechanism, enabling GUI renderers (Electron IPC, Tauri command, WebSocket, postMessage) to be added without changing renderer internals.
- **REQ-114**: The `@selfagency/llm-stream-parser` compatibility shim (FILE-PKG-017) MUST source its renderer re-exports from `@agentsy/renderers` rather than inlining renderer code. All existing subpath exports (`./renderers/*`) MUST continue to work unchanged for shim consumers.
- **REQ-115**: `@agentsy/renderers/browser` MUST accept a `DisplayPort` as its output target in addition to a direct DOM `Element` reference, enabling headless operation in server-side rendering and test environments.
- **REQ-116**: All stream-parsing primitives (thinking, xml-filter, tool-calls, context, structured, formatting, markdown, sse, pipeline, recovery) MUST be accessible without installing any orchestration package. Consumers who only need parsing MAY install `@agentsy/processor` alone (which pulls only `@agentsy/core` + `@agentsy/normalizers`) or continue using `@selfagency/llm-stream-parser`.
- **REQ-117**: The `RendererHandle` interface (`write`, `writeChunk`, `end`) and all shared renderer types (`BaseRendererOptions`, `ThinkingStyle`, `TextOutput`, `OnToolCall`) MUST be exported from the top-level `@agentsy/renderers` (types-only) subpath with no implementation code, enabling downstream packages to declare a `RendererHandle` dependency without pulling in any renderer implementation.
- **REQ-118**: `@agentsy/renderers/vscode` MUST export `createVSCodeAgentLoop` ŌĆö a factory that composes `createAgentLoop` from `@agentsy/agent` with the VS Code renderer. This is the canonical integration point for VS Code extension authors and must be documented as the primary entry point for building VS Code Copilot chat participants.
- **REQ-119**: A future `@agentsy/renderer-gui` package MUST be architecturally supported by the `DisplayPort` abstraction. `DisplayPort` implementations for Electron (IPC), Tauri (command bridge), VS Code Webview (`postMessage`), and WebSocket MUST be expressible as implementors of the same interface without modifying any existing renderer.
- **REQ-120**: A future `@agentsy/extension-vscode` package MUST be expressible as a thin composition of `@agentsy/agent` + `@agentsy/renderers/vscode` + `@agentsy/mcp`. It must require no new primitives beyond what these three packages already provide. The package registers a VS Code chat participant and wires the agent loop to `ChatResponseStream`.
- **REQ-121**: A future `@agentsy/desktop` package MUST be expressible as a composition of `@agentsy/agent` + `@agentsy/renderer-gui` (DisplayPort implementation). It provides a `DesktopAgentApp` factory for Electron and Tauri, wiring the agent loop to the desktop app's display layer. No new streaming or parsing primitives are needed.
- **REQ-122**: `@agentsy/renderers` package.json MUST declare `peerDependencies` for all optional output targets (`ink`, `react`, `cli-markdown`, `streaming-markdown`, `dompurify`) with `peerDependenciesMeta.optional: true`. A consumer installing only `@agentsy/renderers/plain` incurs zero peer dep warnings.
- **REQ-123**: The `DisplayPort` interface MUST support backpressure signalling: a `write(content: DisplayContent): Promise<void>` method that returns a Promise, allowing slow GUI renderers to pause the upstream LLM stream without dropping tokens.
- **REQ-124**: `@agentsy/renderers/ink` MUST export a `DiffViewer` Ink component that renders unified diff (git diff / diff -u format): parses hunk headers (`@@ -a,b +c,d @@`), colours added/removed/context lines, supports keyboard navigation (arrow keys to scroll, spacebar to expand/collapse individual hunks), and handles binary-file markers and no-newline-at-EOF indicators.
- **REQ-125**: `@agentsy/renderers/ink` MUST export a `WorktreeBrowser` Ink component that lists all git worktrees (via `git worktree list --porcelain`), shows branch name, HEAD commit hash, and dirty/clean status for each entry, supports arrow-key navigation, and emits the selected worktree path via an `onSelect` callback on Enter.
- **REQ-126**: `@agentsy/renderers/ink` MUST export a `TextEditor` Ink component providing inline text editing: cursor movement (arrow keys, Ctrl+A/E for line home/end), character insert and delete (Backspace, Delete), multi-line support, a configurable `onChange` callback, and a one-level undo stack (Ctrl+Z, max 50 entries). The component MUST gate all key handling behind an `isFocused` prop so that key events are only consumed when the editor is the active focus target.
- **REQ-127**: `@agentsy/renderers/ink` MUST export a `TerminalPane` Ink component that embeds a pseudo-terminal (pty) via `node-pty` (optional peer dep): spawns a shell or configured command, routes stdin/stdout between the pty and Ink's I/O streams, forwards terminal resize events, and cleans up the pty process on component unmount. A `TerminalPaneFallback` (static scrollable output buffer, zero native deps) MUST also be exported and used automatically when `node-pty` is not installed.
- **REQ-128**: `.github/copilot-instructions.md` MUST be rewritten to describe the `@agentsy` monorepo architecture: `@agentsy` npm org packages, `@selfagency/llm-stream-parser` shim relationship, `packages/` workspace layout, Turborepo pipeline, per-package and root Taskfile commands, Ink component conventions (`useInput` + `isFocused` gating, controlled components), renderer subpath export pattern, dependency boundary rules (CON-024 through CON-029), and testing conventions. The current single-package description is obsolete once the monorepo migration begins.
- **REQ-129**: `@agentsy/renderers/ink` MUST export a `FileBrowser` Ink component that navigates the local filesystem: lists directory contents (files and subdirectories), supports arrow-key navigation, Enter to descend into a directory or call `onSelect(absolutePath)` for a file, Backspace or Left arrow to ascend to the parent. Supports single-select and multi-select modes (Space toggles in multi-select). Controlled: `value?: string` highlights the currently selected path. Zero external deps (Node built-in `fs` only).
- **REQ-130**: `@agentsy/renderers/ink` MUST export a `ChatHistory` Ink component that renders a scrollable list of `ChatMessage` objects (`{ role: 'user' | 'assistant' | 'system' | 'tool'; content: string; timestamp?: Date; isStreaming?: boolean }`). - **REQ-131**: `@agentsy/renderers/ink` MUST export a `Chat` composite Ink component combining `ChatHistory` (top, flex-grow) and `TextEditor` (bottom, fixed height) with a horizontal separator. On Enter (single-line) or Ctrl+Enter (multi-line mode), calls `onSend(text)` and clears the editor. ### Security Requirements

- **SEC-028**: `DisplayPort` implementations that emit content to remote transports (WebSocket, Tauri IPC) MUST sanitize output through `formatXmlLikeResponseForDisplay` before emission to prevent prompt-injection artifacts in rendered output from reaching the GUI layer.
- **SEC-029**: `@agentsy/renderer-gui` (future) MUST enforce a Content Security Policy contract: all HTML content emitted through a `DisplayPort` to a WebView (Electron, Tauri, VS Code Webview) MUST be sanitized with DOMPurify before rendering. Raw `innerHTML` injection is forbidden; only `textContent`, `innerText`, or DOMPurify-cleaned HTML is permitted.
- **SEC-030**: `@agentsy/extension-vscode` (future): the VS Code chat participant handler MUST treat all `request.prompt` input as untrusted external content, passing it through `stripXmlContextTags` / `dedupeXmlContext` before injecting into the agent system prompt, consistent with SEC-013.
- **SEC-031**: `TerminalPane` MUST NOT accept a `command` or `args` prop that derives from untrusted input (LLM output, user-typed text, HTTP request data). The spawned process spec MUST be static configuration only. This mitigates OS command injection (OWASP A03). Any dynamic shell interaction MUST go through stdin to an already-spawned shell, not through a new spawn call.
- **SEC-032**: `ContextPicker` symbol search MUST pass the user-provided query to `rg` or `grep` as a literal argument array (never via shell string interpolation) and spawn with `shell: false`. The query MUST be validated to strip null bytes before use. This mitigates shell injection (OWASP A03) from user-typed search terms.

### Constraints

- **CON-024**: `@agentsy/renderers` and all its subpath packages MUST NOT import from `@agentsy/agent`, `@agentsy/memory`, `@agentsy/mcp`, `@agentsy/runtime`, `@agentsy/connectors`, `@agentsy/scheduler`, `@agentsy/session`, `@agentsy/retrieval`, or any `@agentsy/connector-*` packages. Violation of this boundary is a CI-enforced dependency lint failure.
- **CON-025**: The `DisplayPort` interface MUST be defined in `@agentsy/core` types (or `@agentsy/renderers` if it cannot go in core without circular deps), NOT in any orchestration package, so that both renderers and agent loop implementations can reference it without introducing cycles.
- **CON-026**: `@agentsy/renderers/vscode` MUST NOT import the `vscode` module at the top level. Only structural (duck-typed) interfaces are permitted. This constraint must be enforced by a CI lint rule checking for `import.*from 'vscode'` in the renderer source.
- **CON-027**: `@agentsy/extension-vscode` (future) MAY import `vscode` as a peer dependency, but all VS Code-specific code MUST be isolated to that package. No other `@agentsy/*` package may take `vscode` as a production dependency.
- **CON-028**: Renderer packages are a public API surface. Any breaking change to `RendererHandle`, `BaseRendererOptions`, `DisplayPort`, or `ThinkingStyle` is a semver-major change and requires a migration guide entry in `docs/migration.md`.
- **CON-029**: `node-pty` MUST be declared as an optional peer dependency (with `peerDependenciesMeta["node-pty"].optional: true`) in `packages/renderers/package.json`. `TerminalPane` MUST dynamically import `node-pty` and throw a descriptive error if it is not installed, directing consumers to `pnpm add node-pty`. It MUST NOT be listed in `dependencies` or `devDependencies`.

### Guidelines

- **GUD-021**: When documenting `@agentsy/renderers`, lead with the standalone use case (no agentsy framework). The VS Code renderer documentation should show a 20-line example of using `createVSCodeChatRenderer` inside a VS Code extension with no `@agentsy/agent` dependency. Consumers must be able to discover standalone usage without reading agent documentation.
- **GUD-022**: `DisplayPort` implementations MUST be thin adapters ŌĆö they translate `DisplayContent` tokens to platform calls and nothing else. Business logic, markdown parsing, and thinking-block filtering belong in the renderer layer above the port, not in the port implementation.
- **GUD-023**: For each new renderer target (Electron, Tauri, desktop), provide an integration test that pipes a 50-token mock stream through the full chain (normalizer ŌåÆ processor ŌåÆ renderer ŌåÆ DisplayPort ŌåÆ mock output) and asserts the captured output is byte-for-byte identical to the expected rendered string. This prevents silent regressions in the rendering pipeline.
- **GUD-024**: `@agentsy/extension-vscode` (future) should be designed as a reference implementation first ŌĆö a VS Code extension that installs agentsy as a Copilot chat participant. Its source code should be readable as a tutorial for anyone wanting to build their own VS Code extension on top of `@agentsy/renderers/vscode`.
- **GUD-025**: All interactive Ink components (`DiffViewer`, `WorktreeBrowser`, `TextEditor`, `TerminalPane`, `FileBrowser`, `ChatHistory`, `Chat`, `AgentSelector`, `ModelSelector`, `ModelParamEditor`, `SkillSelector`, `MCPServerManager`, `ContextPicker`, `TokenUsageMeter`) MUST declare `isFocused: boolean` as a required prop and wrap all `useInput` hook calls in a conditional guard: `useInput(handler, { isActive: isFocused })`. This prevents key-capture conflicts when multiple interactive components are mounted simultaneously.
- **GUD-026**: `TextEditor` and `WorktreeBrowser` MUST be fully controlled components: they accept `value` / `entries` and `onChange` / `onSelect` props respectively. Internal state is derived from props, not managed independently. This allows agent orchestration code to programmatically read and set component state without simulating keyboard events.
- **GUD-027**: Composite Ink components that embed two or more interactive child components (`Chat`, `ModelSelector`, `ContextPicker`, `MCPServerManager`) MUST expose a single `isFocused: boolean` prop at the root and route focus internally to exactly one child at a time via a `focusedPanel` state variable. The routing state is internal to the composite; parents see only one `isFocused` prop. This is the single-focus invariant.
- **GUD-028**: Numeric parameter controls in `ModelParamEditor` and any similar stepper components MUST use an arrow-key stepper pattern: Left/Right (or Down/Up) adjusts the value by a configurable `step`, clamped to `[min, max]`. Each control MUST render the current value alongside a dim hint string showing the valid range (e.g., `0.7  [0.0 ŌĆ” 2.0]`). All stepper controls MUST be controlled components (`value: number`, `onChange: (v: number) => void`).
- **REQ-138**: `@agentsy/agent` MUST provide an `AgentTool<TInput, TOutput>` interface with `name: string`, `description: string`, `inputSchema: ZodSchema<TInput>`, and `execute(input: TInput, context: AgentToolContext): Promise<TOutput>`. A `createDefaultTools()` factory MUST return a typed array of all six built-in tools. Each tool MUST be individually importable from `@agentsy/agent/tools/<name>` for tree-shaking. Consumers pass a subset to `createAgentLoop({ tools: [...] })`.
- **REQ-139**: The `fileBrowser` default agent tool MUST wrap the `fd` CLI for fast file discovery: `findFiles({ pattern?, root?, type?, extensions? })` ŌåÆ `Promise<string[]>`. When `fd` is not found in PATH (`ENOENT`), the tool MUST fall back to a recursive `fs.readdir` implementation using `node:fs/promises`. The `fd` binary MUST be invoked with `shell: false` and a literal argument array (CON-031).
- **REQ-140**: The `fileEditor` default agent tool MUST expose `readFile(path)` ŌåÆ `Promise<string>`, `writeFile(path, content)` ŌåÆ `Promise<void>`, and `patchFile(path, find, replace, opts?)` ŌåÆ `Promise<void>`. `patchFile` MUST use the `sd` CLI when available (spawned with `shell: false`), falling back to a built-in `String.replace` / `RegExp` implementation. SEC-033 validation applies before any regex construction. All path arguments are validated per REQ-145.
- **REQ-141**: The `lintFormat` default agent tool MUST invoke `ultracite` via subprocess (`npx ultracite lint` and `npx ultracite format`) with `shell: false`. `lintFiles(paths)` ŌåÆ `Promise<LintDiagnostic[]>` where `LintDiagnostic` is `{ file: string; line: number; col: number; rule: string; message: string; severity: string }`. `formatFiles(paths)` ŌåÆ `Promise<void>`. Both operations MUST throw a descriptive `Error` if `ultracite` is not found (CON-032). No fallback ŌĆö linting without the correct tool produces incorrect output.
- **REQ-142**: The `gitTool` default agent tool MUST use `simple-git` as a regular runtime dependency to expose: `status`, `diff`, `add`, `commit`, `log`, `checkout`, `createBranch`, `push`, `pull`, and `worktreeList`. A `repoPath` argument is accepted per-call or set as a default on the tool instance. `simple-git` errors MUST be re-thrown as descriptive `Error` instances with original message preserved (GUD-029).
- **REQ-143**: The `webBrowser` default agent tool MUST wrap the `agent-browser` CLI via subprocess with `shell: false` and `--json` on every command. Supported operations: `navigate(url)`, `snapshot(opts?)`, `click(ref)`, `fill(ref, text)`, `getText(ref)`, `getUrl()`, `close()`. The tool MUST pass `AGENT_BROWSER_CONTENT_BOUNDARIES=1` in the subprocess environment on every spawn to wrap page content in boundary markers and mitigate prompt injection from untrusted web pages (SEC-033 extension). Throws descriptive `Error` if `agent-browser` is absent from PATH (CON-032).
- **REQ-144**: The `webSearch` default agent tool MUST expose `getWebSearchToolDefinition(opts?)` returning an Anthropic server-side tool object compatible with the Messages API. Supports `web_search_20260209` (default, supports dynamic filtering) and `web_search_20250305` versions, selectable via an `apiVersion` option. Also exports `isWebSearchResult(block)` type guard and `extractWebSearchCitations(response)` ŌåÆ `WebSearchCitation[]` utility. No subprocess; pure data-shaping module with zero runtime side effects and zero peerDeps.
- **REQ-145**: All file paths accepted by `fileEditor` and `fileBrowser` MUST be validated against the configured `workspaceRoot` before any filesystem I/O. Any path that resolves outside the root after `path.resolve()` MUST cause the tool to throw `new Error('Path traversal rejected: path escapes workspace root')`. Validation applies in both the `fd` CLI path and the `node:fs` fallback path (OWASP A01).
- **SEC-033**: `patchFile` MUST validate the `find` string before regex construction: (a) reject strings exceeding 500 characters; (b) apply a static heuristic to reject catastrophic backtracking patterns (deeply nested quantifiers with overlapping character classes); (c) wrap `new RegExp(find)` in a try/catch and propagate a typed error. `find` and `replace` MUST NEVER be interpolated into a shell string ŌĆö they MUST be passed as discrete array elements to `sd` (OWASP A03 injection). `webBrowser` MUST set `AGENT_BROWSER_CONTENT_BOUNDARIES=1` on subprocess env to mitigate prompt injection from untrusted web content (OWASP A03).
- **CON-031**: All default agent tools that spawn subprocesses MUST use `shell: false` with literal argument arrays. String interpolation of user-supplied values into shell command strings is unconditionally forbidden across the agent tools layer. This extends SEC-031 and SEC-032.
- **CON-032**: `fd`, `sd`, `ultracite`, and `agent-browser` MUST be declared as `peerDependencies` with `peerDependenciesMeta.<name>.optional = true` in `packages/agent/package.json`. Each tool wrapper MUST detect at first call whether its CLI dependency is available and throw `new Error('<tool> not found ŌĆö install: <install-hint>')` if absent. Silent wrong-answer failures are forbidden.
- **GUD-029**: `simple-git` MUST be a regular runtime dependency (not optional peer) of `packages/agent`. Git operations are treated as core agent functionality. All other external CLI tool dependencies (`fd`, `sd`, `ultracite`, `agent-browser`) follow CON-032.

## 3. Alternatives

- **ALT-001**: Keep all renderer code inside `@selfagency/llm-stream-parser` and publish only the shim, with no `@agentsy/renderers` package. Rejected: third-party consumers who don't want the `@selfagency` namespace or who are building new packages would have to take a dep on the shim package, which carries the wrong brand and would include all normalizer + processor code they may not need.
- **ALT-002**: One `@agentsy/renderer-<name>` package per renderer (e.g., `@agentsy/renderer-vscode`, `@agentsy/renderer-ink`). Rejected for now: subpath exports in a single `@agentsy/renderers` package achieve identical tree-shaking without exploding the npm publish list. Can revisit if renderer packages diverge significantly in peer dep requirements.
- **ALT-003**: Define `DisplayPort` in `@agentsy/core` instead of `@agentsy/renderers`. Deferred: if the agent loop needs to write directly to a `DisplayPort` (bypassing the renderer layer), it would need `DisplayPort` in core. For now, the renderer layer owns the abstraction; if the agent loop gains direct display capabilities, the interface moves to core.
- **ALT-004**: Build `@agentsy/extension-vscode` as a full VS Code extension with webview UI (not just a chat participant). Deferred to a later plan cycle. A chat participant is simpler, uses `@agentsy/renderers/vscode` directly, and is the right first target. A webview UI would require `@agentsy/renderer-gui` and a `DisplayPort` implementation over `postMessage`, which is planned but not yet implemented.

## 7. Risks & Assumptions

- **RISK-STD-001**: Moving renderer source files from `src/renderers/` to `packages/renderers/` while the `feature/ink-renderer` PR (#46) is open creates a merge conflict surface. Mitigation: complete this migration on a dedicated branch AFTER the ink renderer PR merges. Do not attempt concurrent renderer work on separate branches.
- **RISK-STD-002**: The shim's `./renderers/streaming-md` export path maps to what becomes `./browser` in `@agentsy/renderers`. Renaming the subpath in the shim would break existing consumers. Mitigation: shim maps both `./renderers/streaming-md` AND `./renderers/browser` to the same `@agentsy/renderers/browser` source to maintain backward compat.
- **RISK-STD-003**: `@agentsy/renderers/vscode` currently exports `createVSCodeAgentLoop` which imports from `@agentsy/agent`. This would violate CON-024. Mitigation: `createVSCodeAgentLoop` must move to a separate export (e.g., `@agentsy/renderers/vscode/agent-loop` or into `@agentsy/agent` itself as `createVSCodeAgentLoop`). The renderer-only export (`createVSCodeChatRenderer`) remains in `@agentsy/renderers/vscode` and has no agent dep.
- **RISK-STD-004**: Third-party consumers who pin `@selfagency/llm-stream-parser` at exact version `0.2.0` will receive the shim instead of the original package after the migration. Mitigation: publish the shim at `0.3.0` with a clear CHANGELOG entry; do not publish a `0.2.x` patch that converts to the shim mid-semver.
- **ASSUMPTION-STD-001**: The `vscode` module is only available at runtime inside a VS Code extension process. Assuming all consumers of `@agentsy/renderers/vscode` understand this and provide their own `vscode` runtime context. This must be clearly documented in the package README.
- **ASSUMPTION-STD-002**: The Ink renderer (`packages/renderers/src/ink/`) has no dependency on `@agentsy/agent` in its current form. If this changes (e.g., `createInkAgentLoop` is added to the Ink renderer), it must follow the same separation as `createVSCodeAgentLoop` (RISK-STD-003).
- **RISK-STD-005**: `node-pty` is a native C++ addon requiring compilation at install time (`node-gyp`). It may fail on platforms without build tools (e.g., Windows without Visual Studio Build Tools, Alpine Linux containers, restricted CI environments). Mitigation: `TerminalPaneFallback` provides graceful degradation to a static output buffer; documentation must warn consumers that `TerminalPane` requires a build-capable environment; `TerminalPaneFallback` must be importable independently without triggering the `node-pty` dynamic import.
- **RISK-STD-006**: Ink's `useInput` hook fires for every mounted component that calls it unless `isActive: false` is passed. If `TerminalPane` and `TextEditor` are mounted simultaneously and both have `isFocused: true`, they will both consume the same key events, producing unpredictable behaviour. Mitigation: GUD-025 and GUD-026 establish the `isFocused` gating contract; the parent component using `ink`'s `useFocus` / `useFocusManager` is responsible for ensuring only one interactive component has `isFocused: true` at any time. This must be documented prominently in the `@agentsy/renderers/ink` README.
- **RISK-STD-007**: `ContextPicker` symbol search spawns `rg` or `grep` as subprocesses. These may not be installed in all environments. Mitigation: `rg` is preferred; `grep -rn` is the fallback; if both fail with `ENOENT`, render a graceful empty state with message `"ripgrep not found ŌĆö install ripgrep for symbol search"`. Never throw; always degrade gracefully.
- **RISK-STD-008**: `agent-browser` requires a Chrome installation obtained via `agent-browser install`. In CI or containerised environments, Chrome may be absent or require elevated privileges. Mitigation: `webBrowser` tool throws `new Error('agent-browser requires Chrome ŌĆö run: npx agent-browser install')` if Chrome is not found; consumers can omit `webBrowser` from `createDefaultTools({ include: [...] })` to avoid the dependency in environments where headless browsing is not available.
- **RISK-STD-009**: `fd` and `sd` are Rust CLI binaries distributed via npm. On unusual architectures (e.g., Alpine Linux in Docker, RISC-V), the pre-built binaries may not be available and `cargo install` is required. Mitigation: both tools have robust pure-Node fallbacks (`node:fs/promises` recursion for `fd`, `String.replace` for `sd`). The fallbacks are tested independently (TEST-STD-026, TEST-STD-027) to guarantee correctness parity with the CLI path.
- **ASSUMPTION-STD-003**: `simple-git` wraps the system `git` binary, which must be present in PATH. If `git` is absent, `simple-git` will throw at operation time. This is treated as a hard prerequisite failure, not a graceful fallback scenario, since no reasonable pure-Node git implementation exists as a drop-in replacement.
