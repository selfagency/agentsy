# Plan: Safe Secret Store and .env Handler for Agentsy

## 1. Background Analysis

### Reviewed Resources Summary

- **Walmart Global Tech - Managing Secure API Access to LLMs in Distributed Systems with Dataflow**:
  Discusses centralized, secure access patterns for LLM APIs, emphasizing dataflow governance and credential rotation.

- **IBM - Data Leakage Prevention for LLMs**:
  Focuses on preventing LLMs from leaking sensitive data in prompts, responses, or tool calls, highlighting the need for strict secret handling.

- **Christian Posta - API Keys are a Bad Idea for Enterprise LLM Agents and MCP Access**:
  Argues against long-lived API keys, advocating for short-lived tokens, service accounts, and proper secret management instead of raw keys.

- **Privy - Shamir Secret Sharing**:
  Introduces cryptographic secret sharing (splitting secrets into shares, requiring a quorum to reconstruct), adding resilience against single points of compromise.

- **Cycode - Risks of Hardcoding Secrets in Code Generated by LLMs**:
  Highlights the danger of LLM-generated code containing hardcoded secrets, and the need for tooling to detect and prevent such leaks.

- **NPM keytar**:
  Node.js library that encrypts secrets into an archive (`.keytar`), using the OS keychain to store the encryption key. Provides cross-platform secure storage.

- **GitHub dmno/varlock**:
  Tool/library to "lock" environment variables, preventing processes from overwriting or reading them in unintended ways, adding runtime safety and isolation.

### Key Takeaways

- **Never hardcode secrets.** Use secure storage.
- **Rotate credentials.** Short-lived tokens are preferred.
- **Prevent leakage.** Isolate secrets at runtime (varlock).
- **Encrypt at rest.** Use OS keychain-backed encryption (keytar).
- **Avoid raw API keys.** Use service accounts or token exchanges where possible.
- **Audit for leaks.** Scan code for secrets, especially LLM-generated.

## 2. Requirements for Agentsy Secret Management

Based on the analysis and existing patterns from `opilot`, `z-models-vscode`, and `mistral-models-vscode`, we need:

- **Secure Storage**: Cross-platform encrypted storage for API keys and tokens.
- **VS Code Integration**: Leverage VS Code's `SecretStorage` API for extensions.
- **Fallback Mechanisms**: Allow environment variables (with safety guards) and OS keychain fallback.
- **Validation & Prompting**: Validate keys, prompt users securely, and avoid exposing values in logs.
- **Runtime Isolation**: Prevent accidental leakage or overwriting of secret environment variables (varlock-inspired).
- **Secret Rotation Support**: Allow updating secrets without restart, with notifications.
- **Testing Determinism**: Make secret loading predictable in tests (no side effects).
- **Secret Sharing & Recovery**: Optional Shamir-style sharing for high-security scenarios (future).

## 3. Proposed Architecture

### 3.1 New Package: `@agentsy/secrets`

Create a new monorepo package dedicated to secret management:

- **Core Abstractions**:
  - `SecretStore`: Interface for storing/retrieving/updating/deleting secrets.
  - `SecretValidator`: Optional validator function (regex, length, format).
  - `SecretProvider`: Source of secrets (VS Code, OS keychain, encrypted file, environment).

- **Implementations**:
  - `VSCodeSecretStore`: Wraps `ExtensionContext.secrets`.
  - `KeytarSecretStore`: Uses `keytar` to encrypt/decrypt an archive.
  - `EnvironmentSecretProvider`: Loads from `process.env` (with varlock-style locking).
  - `CompositeSecretProvider`: Chains multiple providers (e.g., VS Code first, then env, then keytar).

- **Varlock Integration**:
  - Implement `VariableLocker` API that wraps environment variable access:
    - `getLocked(key)`: Reads a variable, marks it as read-locked.
    - `setLocked(key, value)`: Writes a variable, marks it as write-locked.
    - Prevents accidental overwriting or reading of locked variables across the process.
  - Use `WeakMap` or `Proxy` to enforce locks at runtime.

- **API Key Manager** (extracted from `opilot` and generalized):
  - `ApiKeyManager`: Encapsulates prompting, validation, caching, and change events.
  - Methods: `initialize()`, `getApiKey()`, `setApiKey()`, `onDidChangeApiKey()`.
  - Configurable: secret keys, display names, prompt messages, validation functions.
  - Supports both interactive (window.showInputBox) and non-interactive (env) modes.

### 3.2 Safe `.env` Handler

- **`AgentsyEnvLoader`**:
  - Loads `.env` files using `dotenv` (or similar).
  - Applies varlock-style locking to all loaded variables to prevent tampering.
  - Supports `.env.local`, `.env.development`, `.env.production` with precedence rules.
  - Validates that required variables are present and match expected patterns.

- **CLI Tool** (e.g., `agentsy env`):
  - Commands:
    - `agentsy env init`: Generate a secure `.env` template.
    - `agentsy env lock`: Mark all variables as locked (create a lockfile).
    - `agentsy env unlock`: Remove locks (dangerous, requires confirmation).
    - `agentsy env encrypt`: Encrypt the current `.env` into `.env.keytar` (using keytar).
    - `agentsy env decrypt`: Decrypt `.env.keytar` back to plaintext (prompts for keychain unlock).

### 3.3 Secret Rotation and Auditing

- **Rotation Hooks**: Allow registering a function that runs when a secret is updated.
- **Audit Logging**: Emit events (via a logger) when secrets are accessed/changed, without exposing values.
- **Leak Detection Scanner**: Optional tool to scan code for hardcoded secrets, using `keytar` patterns or common regexes.

## 4. Implementation Plan

### Phase 1: Core `@agentsy/secrets` Package

1. Initialize `packages/secrets` with dependencies: `keytar`, `dotenv`, `vscode` (peer optional).
2. Define TypeScript interfaces and base classes for `SecretStore`, `SecretProvider`, `VariableLocker`.
3. Implement `VSCodeSecretStore` and `KeytarSecretStore`.
4. Implement `VariableLocker` using a `Proxy` to trap `process.env` access and enforce locks.
5. Implement `ApiKeyManager` with validation, prompting, and caching (extract from `opilot` patterns).
6. Add comprehensive tests:
   - Unit tests for each store type.
   - Integration tests for varlock behavior (attempt to overwrite locked vars).
   - Tests for secret rotation and change events.

### Phase 2: `.env` Handler and CLI

1. Implement `AgentsyEnvLoader` using `dotenv` and integrating `VariableLocker`.
2. Add CLI commands via `yargs` or `commander`:
   - `agentsy env init`, `lock`, `unlock`, `encrypt`, `decrypt`.
3. Implement `keytar` integration for encrypting/decrypting `.env` files.
4. Add documentation for `.env` best practices:
   - Never commit `.env`.
   - Use `.env.example` as template.
   - Lock sensitive variables in production.
5. Test CLI commands in isolated environments.

### Phase 3: Adapter to `@agentsy/vscode`

1. Update `@agentsy/vscode` to expose `ApiKeyManager` from `@agentsy/secrets`.
2. Provide a helper `createApiKeyManager(context, options)` that instantiates the right stores.
3. Ensure `ApiKeyManager` integrates with VS Code's `SecretStorage` and prompts.
4. Add example usage in production-provider.md.

### Phase 4: Security Enhancements

1. Add optional Shamir secret sharing utilities (using `shamir-secret-sharing-firebase` or similar) for high-security use cases.
2. Implement a comprehensive leak detection scanner with entropy-based detection:
   - **Phase 1**: Simple regex-based scanner for common secret patterns (API keys, tokens, secrets).
   - **Phase 2**: Entropy-based scanner adapted from `vscode-entropy-scanner`:
     - Shannon entropy calculation for base64 and hex strings.
     - Thresholds: 4.5+ for base64, 3.0+ for hex strings.
     - Minimum length: 20+ characters (configurable).
     - Signature-based exclusion system (blake2s) to suppress false positives.
     - VS Code integration: code action provider, diagnostic reporting.
   - **Phase 3**: Composite scanner combining regex and entropy detection.
3. Add audit logging to `ApiKeyManager` (log accesses and changes without values).
4. Document security model: encryption at rest, runtime isolation, rotation, audit.

### Phase 5: Documentation & Migration

1. Create `packages/secrets/README.md` with API docs and security guidelines.
2. Add migration guide for moving from direct `context.secrets` usage to `ApiKeyManager`.
3. Update getting-started.md to include secret management steps.
4. Publish `@agentsy/secrets` to npm (private or public as appropriate).

## 5. Risk Mitigation & Best Practices

- **Never Log Secrets**: Ensure all logging and error messages omit secret values (use asterisks).
- **Prevent Clipboard Leaks**: When prompting users, avoid storing values in clipboard unnecessarily.
- **Short-lived Tokens**: Encourage token-based auth (JWT, session tokens) over static API keys where possible.
- **Secure Defaults**: Default to secure behavior (e.g., reject weak keys, require explicit opt-in for env fallbacks).
- **Environment Isolation**: Use varlock to prevent process-wide leakage.
- **Audit Regularly**: Provide CLI tool to scan for leaked secrets in code.

---

## 6. Summary

By synthesizing `keytar`, `varlock`, and the API key management patterns from `opilot`, we will deliver a robust, secure secret management solution for the Agentsy framework. The new `@agentsy/secrets` package will provide:

- Encrypted storage (keytar)
- Runtime variable isolation (varlock)
- VS Code integration
- Safe `.env` handling
- Secret rotation and auditing

This aligns with industry best practices and reduces the risk of secret leakage in LLM-powered applications.
